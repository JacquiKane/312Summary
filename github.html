Topic: Git/GitHub

Basic Configuration

$ git config --global user.name "Annie Student"

$ git config --global user.email annie_student@you_who.com

The user name and e-mail are then associated with your account.

Creating Your GitHub Repository

We have you login to GitHub, using your login credentials, and create a new repository. You are given Git commands to type in the command line, where anniestudent/xyz is the reference to your repository.

First, navigate to your folder you will use for the source code you will be putting in the repository ($ cd myProjectFolder), type

$ git init

$ git remote add origin https://github.com/anniestudent/xyz

$ git branch -M main

$ git push -u origin main

Staging Files

Before you add files to the GitHub repository, you have to let Git know about them. To do this you stage the files (Git's term) or put them in the staging area, by using:

$ git add . or git add --All or git add -A or $ git add *.js

This puts a set of files in a holding state, before being pushed to the GitHub repository. Adding the files to the staging area is like queuing up a set of changes to be committed.

Committing Files
Save in the GitHub repository permanently.

$ git commit -m "some text describing the commit"

Then, push the files to the GitHub repository,

$ git push -u origin main

3 possible states for any file in the folder:

State Details
Staged - ready to be committed - all necessary changes made
Unmodified - No changes made to file
Modified - some changes made, not yet saved
Committed - all changes made and saved in local repository

Reveal the status of any file in the folder, $ git status

Ignore Files
Ignore certain files.

create a file called .gitignore, and add any files or folders you do not want to be included in the project files - then, when a

$ git add --All

command is issued, these files will not be included in the staging area, will not be committed, and will not be stored in the GitHub repository.

Branching

A branch is a snapshot of the code from the last commit. To create a branch, with a title of branchA from the command line, type the following:

$ git checkout -b branchA

or

$ git switch -c branchA

This creates a new and independent copy of the code in the last commit, whatever that was. You can then add files to this branch, but not to main.

To add the files to the branchA branch,

$ git push origin branchA

Switch back to the main branch by typing,

$ git checkout main
or
$ git switch main

See all and every branch by typing

$ git branch

Download a copy of a branch, like main

$ git clone main
Merging

To merge the changes made in say, branchA, you would switch back into the main branch and then run the command,

$ git merge branchA
Git Flow

The typical development sequence for a branch is,

  • create the branch ( git checkout...) and add the code
  • make changes to the branch, and test
  • merge the changes with the main branch (switch to main, then use git merge)
  • get rid of the branch, delete the branch (use git branch -d ...)
